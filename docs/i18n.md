# Internationalization (i18n)

The framework provides a lightweight i18n system based on React context. Translation keys in page configs are automatically resolved at render time.

> The [admin-console](../examples/admin-console/) example uses literal strings for simplicity. This guide shows how to add multi-language support on top of that pattern.

## Setup

Wrap your app in `I18nProvider` with the current locale and a translations map. The admin-console example does this in `src/main.tsx`:

```tsx
import { I18nProvider } from 'pageforge';

const translations = {
  en: {
    'list.loading': 'Loading resources...',
    'list.emptyTitle': 'No projects',
    'list.emptyDescription': 'Create a project to get started.',
  },
};

createRoot(document.getElementById('root')!).render(
  <BrowserRouter>
    <I18nProvider locale="en" translations={translations}>
      <App />
    </I18nProvider>
  </BrowserRouter>
);
```

### Props

| Prop | Type | Description |
|------|------|-------------|
| `locale` | `string` | Current locale key (e.g. `'en'`, `'es'`, `'pt-BR'`) |
| `translations` | `Record<string, Translations>` | Map of locale → key-value translation pairs |
| `children` | `React.ReactNode` | App content |

`Translations` is typed as `Record<string, string>`.

## How It Works in the Example

The admin-console example uses **literal strings** in page configs:

```typescript
// src/configs/projects.ts
columns: [
  { key: 'name', label: 'Name', sortable: true },
  { key: 'owner', label: 'Owner', sortable: true },
]
```

Framework components pass every label through `t(label, label)`. Since `'Name'` isn't a key in the translations map, `t` returns the fallback — which is the literal string `'Name'`. This means i18n works transparently even when you're not using translation keys.

The example only defines translations for framework-level strings that override defaults:

```typescript
const translations = {
  en: {
    'list.loading': 'Loading resources...',    // overrides default "Loading..."
    'list.emptyTitle': 'No projects',          // overrides default "No items"
    'list.emptyDescription': 'Create a project to get started.',
  },
};
```

## Adding Multi-Language Support

To convert the example to multi-language, replace literal strings with translation keys in your configs:

**Before (literal strings):**

```typescript
{ key: 'name', label: 'Project name', type: 'text', placeholder: 'Enter project name' }
```

**After (translation keys):**

```typescript
{ key: 'name', label: 'field.project.name', type: 'text', placeholder: 'field.project.name.placeholder' }
```

Then define translations for each locale:

```typescript
// src/i18n/en.ts
export const en: Record<string, string> = {
  'page.projects.title': 'Projects',
  'page.projects.description': 'Manage your projects',
  'col.project.name': 'Name',
  'col.project.owner': 'Owner',
  'col.project.status': 'Status',
  'action.project.create': 'Create project',
  'action.project.delete': 'Delete',
  'field.project.name': 'Project name',
  'field.project.name.placeholder': 'Enter project name',
  'list.loading': 'Loading resources...',
  'list.emptyTitle': 'No projects',
  'list.emptyDescription': 'Create a project to get started.',
  'form.cancel': 'Cancel',
};
```

```typescript
// src/i18n/es.ts
export const es: Record<string, string> = {
  'page.projects.title': 'Proyectos',
  'page.projects.description': 'Administre sus proyectos',
  'col.project.name': 'Nombre',
  'col.project.owner': 'Propietario',
  'col.project.status': 'Estado',
  'action.project.create': 'Crear proyecto',
  'action.project.delete': 'Eliminar',
  'field.project.name': 'Nombre del proyecto',
  'field.project.name.placeholder': 'Ingrese el nombre del proyecto',
  'list.loading': 'Cargando...',
  'list.emptyTitle': 'Sin proyectos',
  'list.emptyDescription': 'Cree un proyecto para comenzar.',
  'form.cancel': 'Cancelar',
};
```

Aggregate them in an index:

```typescript
// src/i18n/index.ts
import { en } from './en';
import { es } from './es';

export const TRANSLATIONS: Record<string, Record<string, string>> = { en, es };
```

Then update `main.tsx`:

```tsx
import { TRANSLATIONS } from './i18n';

<I18nProvider locale={locale} translations={TRANSLATIONS}>
```

## Translation Files

Create one file per locale, each exporting a flat `Record<string, string>`. Keep all locales in a `src/i18n/` directory:

```
src/i18n/
├── en.ts
├── es.ts
├── pt-BR.ts
└── index.ts
```

To add a new language, create the file and register it in `index.ts`. All framework components pick it up automatically when the `locale` prop changes.

## Using Translations in Custom Components

The `useI18n` hook provides the `t` function and current `locale`:

```tsx
import { useI18n } from 'pageforge';

function ProjectHeader() {
  const { t, locale } = useI18n();
  return <h1>{t('page.projects.title', 'Projects')}</h1>;
}
```

### `t(key, fallback?)`

| Parameter | Type | Description |
|-----------|------|-------------|
| `key` | `string` | Translation key to look up |
| `fallback` | `string` (optional) | Returned if no translation exists. Defaults to the key itself. |

Lookup order:
1. `translations[locale][key]` — if found, return it
2. `fallback` — if provided, return it
3. `key` — return the raw key string

## Key Naming Convention

Use dot-separated prefixes to organize translation keys:

| Prefix | Usage | Example |
|--------|-------|---------|
| `page.*` | Page titles and descriptions | `page.projects.title` |
| `col.*` | Table column headers | `col.project.name` |
| `action.*` | Button labels | `action.project.create` |
| `field.*` | Form field labels | `field.project.name` |
| `option.*` | Select option labels | `option.project.status.active` |
| `nav.*` | Navigation items | `nav.projects` |
| `list.*` | Framework list defaults | `list.loading` |
| `form.*` | Framework form defaults | `form.cancel` |
